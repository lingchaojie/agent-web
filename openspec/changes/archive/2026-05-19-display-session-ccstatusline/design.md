## Context

The current web client already has session selection, live WebSocket subscription, structured render-state updates, and a native-Claude-like conversation surface. It exposes activity, connection, transcript source, lifecycle, and structured render state, but it does not display the local Claude Code statusline generated by the configured `statusLine.command`.

The user's local Claude Code configuration uses a command-based statusline:

- `command`: `npx -y ccstatusline@latest`
- `padding`: `0`
- `refreshInterval`: `10`
- dark theme

The target UX is the compact Claude Code client statusline shown in the user's screenshot: black terminal-like background, ANSI-colored text, natural command-provided line breaks, and no web-specific field reconstruction. The panel belongs above the composer, updates when the selected session changes, and refreshes live for the selected session.

## Goals / Non-Goals

**Goals:**

- Execute the configured local `statusLine.command` on the server with stdin context for the relevant app session.
- Maintain session-scoped statusline state so each selected session has an independent latest output.
- Push statusline snapshots and updates over the existing session WebSocket stream.
- Render the output in the web client as a compact terminal-like `<pre>` panel above the composer, preserving ANSI colors.
- Refresh on the configured `statusLine.refreshInterval`, defaulting safely when the setting is missing.
- Remove local-only keyboard shortcut hints such as `(shift+tab to cycle)` before showing web output.
- Keep statusline output out of durable conversation transcript blocks.

**Non-Goals:**

- Reimplement `ccstatusline` fields in React.
- Add user-editable statusline configuration in the web UI.
- Persist historical statusline samples in transcripts.
- Make historical, non-running Claude transcript entries actively refresh without an app session/process context.
- Change existing conversation render-region semantics.

## Decisions

### Reuse the local statusLine command instead of reconstructing fields

The server will read Claude Code settings and execute the configured command. This preserves the user's local statusline behavior, colors, model display, git/worktree information, usage formatting, and future `ccstatusline` changes without duplicating that logic in this app.

Alternative considered: derive model, context, git, usage, and permission state from existing session state and render custom React chips. That was rejected because the user explicitly wants the web output to match the local cc client and not be split into many rows or fields.

### Treat statusline as session stream state

Add a session-scoped statusline value to the existing realtime stream contract. Initial WebSocket subscribe sends the latest known statusline with the snapshot, and later refreshes send ordered `statusline-changed` events. The client reducer stores the latest statusline separately from conversation blocks and render regions.

Alternative considered: add a polling REST endpoint per selected session. That would work but duplicates realtime connection lifecycle handling and makes session switching/reconnect recovery less consistent with existing stream behavior.

### Refresh statusline only while sessions have interested clients

The realtime hub should start or keep a refresh timer for a session while at least one client is subscribed and stop it when the last client detaches. Refresh cadence follows `statusLine.refreshInterval`; if no value is configured, use a small safe default such as 10 seconds. Manual refresh can also occur on subscribe so the user sees current output promptly.

Alternative considered: run refresh loops for all known sessions. That wastes process launches and can execute user-configured commands for sessions nobody is viewing.

### Pass Claude Code-like stdin context to the command

`ccstatusline` expects JSON on stdin and exits with `No input received` without it. The server will construct per-session input from app session data: session id, native Claude session id when known, project/worktree path, model/effort/settings when available, activity/lifecycle, and transcript/render metadata that maps to the Claude Code statusline input contract. Missing fields should be omitted or set to safe defaults rather than invented as user-visible facts.

### Render ANSI output, not semantic fields

The client will convert ANSI SGR sequences to safe HTML spans or equivalent React elements and place them in a single terminal-like statusline panel. CSS uses a dark background, monospace font, compact padding, `white-space: pre`, and horizontal overflow. The UI does not split the command output into chips or separate rows beyond the command's own newlines.

### Sanitize local-only shortcut hints

Before broadcasting or rendering, remove known native-only hints such as `(shift+tab to cycle)` because they are misleading in the web UI. This filtering is presentation cleanup only; it must not alter transcript content because statusline output is never transcript content.

## Risks / Trade-offs

- **Command execution cost** → Run statusline commands only for subscribed sessions and respect the configured refresh interval.
- **Command failure or missing stdin fields** → Return compact statusline error state with timestamp and keep the previous successful output until replaced, without appending transcript content.
- **ANSI rendering XSS risk** → Escape text before applying ANSI styling and never inject raw command output as HTML.
- **Mismatch with real Claude Code statusline input contract** → Keep the statusline input builder isolated and covered by tests so fields can be adjusted without touching UI rendering.
- **Multiple browser clients for one session** → Share one refresh loop and broadcast the same session-scoped update to all subscribers.
- **Stopped or historical sessions** → Show the latest cached statusline or a stopped/unavailable state; do not keep refreshing when there is no meaningful live session context.
